# AI 能力融入 Edict 三省六部架构

## 原作者核心流程（保持不变）

```
皇上(User) 
  ↓
太子(Taizi) - 分拣：旨意 vs 闲聊
  ↓（建立任务）
中书省(Zhongshu) - 规划：分析需求、拆解子任务
  ↓
门下省(Menxia) - 审议：审查方案、准奏/封驳（最多3轮）
  ↓（准奏）
尚书省(Shangshu) - 派发：分配给六部
  ↓
六部并行执行（礼/户/兵/刑/工/吏）
  ↓
尚书省汇总 → 中书省回奏 → 皇上
```

**状态机**：Pending → Taizi → Zhongshu → Menxia → Assigned → Doing → Review → Done

---

## AI 增强点（不改动流程，只增强能力）

### 1. 太子分拣 + AI 意图识别

**原流程**：太子根据关键词判断

**AI增强**：
```
用户："帮我看看这个bug"
  ↓
太子AI：理解意图 → 这是代码审查任务
  ↓
太子：建立任务 → 转交中书省
```

**实现**：太子增加 `ai_intent` 技能，自然语言理解

---

### 2. 中书省规划 + AI 自动拆解

**原流程**：中书省人工分析需求、拆解子任务

**AI增强**：
```
中书省接到："添加用户登录功能"
  ↓
中书AI自动拆解：
  1. 兵部：创建 auth.ts 核心代码
  2. 户部：设计用户表结构  
  3. 礼部：写 API 文档
  4. 刑部：安全审查
  5. 工部：部署配置
  ↓
生成完整方案 → 提交门下省审议
```

**实现**：中书省增加 `ai_planning` 技能

---

### 3. 门下省审议 + AI 自动审查

**原流程**：门下省人工审查方案，准奏或封驳

**AI增强**：
```
门下省接到中书方案
  ↓
门下AI自动审查：
  ✓ 代码规范检查
  ✓ 潜在bug扫描
  ✓ 安全漏洞检测
  ✓ 可行性评估
  ↓
生成审查报告：
  - 问题1：缺少错误处理 → 封驳
  - 问题2：性能可优化 → 建议
  ↓
门下省：根据AI报告决定准奏/封驳
```

**实现**：门下省增加 `ai_review` 技能

**封驳循环**：
```
中书省 → 门下省AI审查 → 发现问题 → 封驳
    ↑                                    ↓
    └──────── 修改后重新提交 ←───────────┘
         （最多3轮，AI记录问题）
```

---

### 4. 尚书省派发 + AI 智能调度

**原流程**：尚书省根据规则派发六部

**AI增强**：
```
尚书省接到准奏方案
  ↓
尚书AI分析：
  - 任务A需兵部，预计15分钟
  - 任务B需户部+兵部，预计30分钟
  - 兵部当前忙碌，建议延后
  ↓
智能调度：
  先执行A，B并行准备
  预测整体完成时间
```

**实现**：尚书省增加 `ai_schedule` 技能

---

### 5. 六部执行 + AI 代码生成

**原流程**：六部人工执行各自任务

**AI增强（按部门）**：

#### 兵部（代码实现）
```
兵部接到："创建登录接口"
  ↓
兵部AI：
  1. 读取现有代码风格
  2. 生成 auth.ts 代码
  3. 自动生成单元测试
  4. 自我验证编译通过
  ↓
提交尚书省汇总
```
**技能**：`ai_code_gen`、`ai_test_gen`

#### 礼部（文档编制）
```
礼部接到："写API文档"
  ↓
礼部AI：
  1. 读取代码注释
  2. 自动生成 API.md
  3. 生成调用示例
```
**技能**：`ai_doc_write`

#### 户部（数据分析）
```
户部接到："设计用户表"
  ↓
户部AI：
  1. 分析业务需求
  2. 生成数据库模型
  3. 优化索引设计
```
**技能**：`ai_data_model`

#### 刑部（测试审查）
```
刑部接到："审查登录功能"
  ↓
刑部AI：
  1. 安全扫描
  2. 边界测试
  3. 生成测试报告
```
**技能**：`ai_security_scan`

#### 工部（基础设施）
```
工部接到："部署配置"
  ↓
工部AI：
  1. 生成 Dockerfile
  2. 生成 K8s 配置
  3. 生成 CI/CD 脚本
```
**技能**：`ai_deploy_script`

---

### 6. 尚书省汇总 + AI 自动整合

**原流程**：尚书省人工收集六部结果

**AI增强**：
```
六部执行完成
  ↓
尚书AI：
  1. 收集各部输出
  2. 检查冲突
  3. 自动合并代码
  4. 生成汇总报告
  ↓
转 Review 状态 → 回调中书省
```

**实现**：尚书省增加 `ai_merge` 技能

---

### 7. 中书省回奏 + AI 报告生成

**原流程**：中书省人工写回奏报告

**AI增强**：
```
尚书省汇总完成
  ↓
中书AI：
  1. 读取完整执行记录
  2. 生成回奏报告：
     - 已完成：用户登录功能
     - 涉及文件：auth.ts, user.ts...
     - 测试结果：通过
     - 部署状态：已上线
  3. 生成给用户的消息
  ↓
状态转为 Done → 回复皇上
```

**实现**：中书省增加 `ai_report` 技能

---

## 完整 AI 增强流程示例

### 场景：用户说"添加用户登录功能"

```
【太子分拣】+ AI意图识别
用户："添加用户登录功能"
太子AI：理解 → 这是功能开发任务
太子：建立任务 #123 → 转中书省
状态：Pending → Taizi

【中书省规划】+ AI自动拆解
中书AI分析：
  - 需要：JWT认证、密码加密、会话管理
  - 拆解为5个子任务
中书省：生成方案 → 提交门下省
状态：Taizi → Zhongshu

【门下省审议】+ AI自动审查
门下AI审查：
  ✓ 技术方案可行
  ⚠ 建议增加密码强度校验
门下省：准奏（附建议）→ 转尚书省
状态：Zhongshu → Menxia → Assigned

【尚书省派发】+ AI智能调度
尚书AI调度：
  - 兵部：立即执行（高优先级）
  - 户部：并行执行
  - 礼部：等兵部完成后执行
状态：Assigned → Doing

【六部并行执行】+ AI代码生成
兵部AI：生成 auth.ts（含JWT实现）
户部AI：设计用户表（含密码字段）
礼部AI：等待中...
刑部AI：等待中...
工部AI：等待中...

【尚书省汇总】+ AI自动整合
兵部完成 ✓
户部完成 ✓
礼部AI：根据代码生成文档 ✓
刑部AI：自动安全扫描 ✓
工部AI：生成部署配置 ✓
尚书AI：合并所有输出
状态：Doing → Review

【中书省回奏】+ AI报告生成
中书AI生成报告：
  "用户登录功能已完成：
   - JWT认证已实现
   - 密码bcrypt加密
   - API文档已生成
   - 安全扫描通过"
状态：Review → Done

【太子回复】
太子：发送回奏给用户
```
